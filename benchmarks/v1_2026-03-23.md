## Análise Qualitativa

### ClassicNLP (TF-IDF + Logistic Regression)

**Resultado no benchmark: 0/12 acertos de tag — sempre retornou `NÃO IMPORTANTE`**

Este foi o achado mais crítico do benchmark. O modelo classificou **todos os 12 casos** como `Improdutivo / NÃO IMPORTANTE`, tornando-o inútil como classificador neste estado. A causa raiz é o **threshold de confiança de 0.6** combinado com o **dataset de treino pequeno (~56 exemplos)**: o modelo raramente atinge 60% de confiança em qualquer classe que não seja a mais frequente no treino, caindo sempre no fallback.

A sumarização extrativa funcionou corretamente em vários casos — o modelo selecionou sentenças representativas do corpo do email. Porém, sem classificação correta, a funcionalidade é irrelevante na prática.

**Pontos fortes:**
- Latência de 8–133ms (média 21ms) — mais de 75x mais rápido que o Claude
- Totalmente offline, sem dependência de API externa
- Determinístico e reproduzível — mesma entrada sempre gera mesma saída
- Sumarização extrativa TF-IDF funcionou bem

**Falhas observadas:**
- Reunião corporativa → `NÃO IMPORTANTE` (esperado: REUNIÃO)
- Reclamação formal → `NÃO IMPORTANTE` (esperado: RECLAMAÇÃO)
- Phishing bancário → `NÃO IMPORTANTE` (acertou category Improdutivo, errou tag)
- Email urgente → `NÃO IMPORTANTE` (esperado: URGENTE)
- Email em inglês → `NÃO IMPORTANTE` (falha de generalização linguística)

**Diagnóstico:** threshold de 0.6 é alto demais para 56 exemplos de treino em 8 classes (~7 exemplos/classe). Para funcionar, precisaria de pelo menos 200–500 exemplos por classe ou redução do threshold para ~0.35.

**Técnicas NLP aplicadas:**
Detecção de idioma (langdetect) → lowercasing → tokenização por regex → remoção de stopwords (NLTK multilíngue) → stemming (SnowballStemmer) → TF-IDF com unigramas e bigramas (500 features) → features manuais (comprimento normalizado, presença de `?`, action words) → Logistic Regression → threshold de confiança com fallback → sumarização extrativa por score médio TF-IDF

### HuggingFace (mDeBERTa-v3-base-mnli-xnli, zero-shot)

**Resultado no benchmark: 7/12 acertos de category, 6/12 acertos de tag**

Desempenho intermediário. Acertou os casos mais claros (reclamação, solicitação, urgente, informativo, comunicado) mas falhou em padrões semânticos sutis.

**Falhas notáveis:**
- **Spam → URGENTE:** o modelo foi enganado pelos marcadores de urgência artificiais ("OFERTA IMPERDÍVEL!!!", "só até meia-noite") — incapaz de distinguir urgência real de urgência de marketing
- **Phishing → URGENTE:** idem — "AVISO URGENTE DO SEU BANCO" ativou o label URGENTE em vez de POSSÍVEL GOLPE
- **Agradecimento → RECLAMAÇÃO:** falha grave — uma mensagem de agradecimento foi classificada como reclamação, possivelmente pelo vocabulário emocional
- **Email em inglês → NÃO IMPORTANTE:** apesar do modelo ser multilíngue, não reconheceu a solicitação em inglês
- **Mensagem curta → POSSÍVEL GOLPE:** texto de 5 palavras gerou classificação aleatória (baixa confiança não ativou threshold corretamente)
- **Boas-vindas → REUNIÃO:** não distinguiu "email que menciona 1:1" de "convite de reunião"

**Pontos fortes:**
- Zero-shot funciona sem dados de treino — acertou reclamação, solicitação e urgente de forma independente
- Latência 1.3–5s (média 1.6s) — aceitável para uso assíncrono
- Sem custo por chamada além do token HuggingFace

**Técnicas NLP aplicadas:**
Transformer DeBERTa-v3 (multilíngue, 278M parâmetros) → cabeça NLI (Natural Language Inference) → hipótese "Este email é sobre {label}" para cada tag → ranking por probabilidade de entailment → threshold de confiança 0.4 → sumarização por `_first_sentence` (extração simples, sem semântica)

---

### Claude (claude-sonnet-4-20250514)

**Resultado no benchmark: 10/12 acertos de category, 11/12 acertos de tag**

Melhor desempenho geral. Único modelo que classificou corretamente os casos mais difíceis: phishing, spam, email em inglês e boas-vindas (com auxílio do recovery de tag↔categoria).

**Achados notáveis:**
- **Boas-vindas/onboarding:** retornou `Produtivo/INFORMATIVO` — inconsistência capturada pelo recovery code, que corrigiu para `Improdutivo/INFORMATIVO`. Sem o recovery, o resultado seria inválido
- **Mensagem curta "ok, pode fechar o chamado":** classificou como `Produtivo/SOLICITAÇÃO` — interpretação defensável ("fechar chamado" é uma ação), mas o esperado seria `NÃO IMPORTANTE`
- **Email em inglês:** único modelo a classificar corretamente como `Produtivo/SOLICITAÇÃO`
- **Resumos:** os mais contextualizados e informativos dos três modelos — gerados pelo próprio LLM a partir do conteúdo real

**Limitações confirmadas:**
- Latência 2.3–4.5s (média 3.4s) — a mais alta dos três
- Custo por chamada em tokens
- Inconsistência ocasional entre category e tag (observada em boas-vindas) — requer validação no post-processing

**Técnicas NLP aplicadas:**
Prompt engineering com definições de categoria + 4 exemplos few-shot + formato JSON estruturado → geração de `category`, `tag`, `summary` e `suggested_response` contextualizados → validação de consistência tag↔category com mapeamento determinístico no post-processing

## Recomendação de Uso

| Cenário | Modelo Recomendado | Motivo |
|---------|-------------------|--------|
| Produção com precisão crítica | **Claude** | Melhor acurácia, resumos e respostas contextualizadas |
| Volume alto / baixo custo | **HuggingFace** | Sem custo por token, acurácia razoável em casos claros |
| Offline / latência mínima | **ClassicNLP** | Requer expansão do dataset para ser viável |
| Pipeline híbrido | **ClassicNLP triagem + Claude** | ClassicNLP filtra casos óbvios; Claude resolve ambíguos |

> **Nota:** O ClassicNLP necessita de expansão do dataset de treino (≥200 exemplos/classe) e revisão do threshold de confiança antes de ser adequado para produção.
